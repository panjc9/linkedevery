## 一、模型总览

### 1.1 核心理念

> **不是"找相同"，而是"诊断→处方→配药"**

企业需求有四个等级，每个等级对应不同的 TRL/IRL 匹配规则。匹配的核心是找到**"导师 TRL 领先企业 IRL 最佳带教距离"**。

### 1.2 四等级 × TRL/IRL 总览

```
┌─────────────────────────────────────────────────────────────────┐
│  L4 研究型  "这个方向我们不懂，需要导师带我们从0开始"             │
│  企业 IRL: L1-L2 | 导师最佳 TRL: T3-T5 | 最佳差距: +2~+4       │
│  定价系数: 3.0 | 周期: 6-24月 | 导师: 学术型                     │
├─────────────────────────────────────────────────────────────────┤

│  定价系数: 2.0 | 周期: 1-6月 | 导师: 技术型                      │
├─────────────────────────────────────────────────────────────────┤
│  L2 工程型  "方案都有了，需要人帮我们落地实施"                    │
```

## 二、四等级详细定义

### 2.1 L4 研究型需求

**核心问题：** "这个方向我们不懂，需要导师带我们从0开始"

| 维度 | 定义 |
|------|------|
| **需求特征** | 前沿技术探索、基础研究、新领域进入、技术可行性论证 |
| **典型场景** | "我们想引入AI，但不知道从哪开始" / "这个新技术能用在我们行业吗" |
| **企业 IRL** | L1-L2（概念→原型） |
| **技术 TRL** | T1-T3（基础研究→概念验证） |

**核心匹配规则：**

```
导师 TRL 最佳区间：T3-T5
最佳 TRL-IRL 差距：+2 ~ +4

匹配逻辑：
• 企业 IRL=1（概念），导师 TRL=3~5 → ✅ 最佳
• 企业 IRL=2（原型），导师 TRL=4~5 → ✅ 最佳
• 企业 IRL=1（概念），导师 TRL=6+ → ⚠️ 勉强（太超前）
• 企业 IRL=1（概念），导师 TRL=1~2 → ❌ 不匹配（太弱）

特殊规则：
• 研究型需求允许更大 TRL 差距（+4 仍可用）
• 但需要导师有"翻译能力"（学术→产业的语言转换）
• 优先匹配有产学研合作经验的导师
```

| 属性 | 值 |
|------|-----|
| 导师类型 | 学术型（教授/研究员） |
| 服务形式 | 技术咨询、联合研究、人才培养 |
| 周期 | 6-24个月 |
| 定价系数 | 3.0（稀缺性+不确定性高） |

---

### 2.2 L3 技术型需求

**核心问题：** "我们知道要做什么，但技术上搞不定"

| 维度 | 定义 |
|------|------|
| **需求特征** | 技术攻关、算法优化、系统架构、技术选型、原型开发 |
| **典型场景** | "我们需要一个质检模型，但自己搞不定" / "系统架构需要重新设计" |
| **企业 IRL** | L2-L4（原型→推广） |
| **技术 TRL** | T4-T6（实验室验证→中试） |

**核心匹配规则：**

```
导师 TRL 最佳区间：T5-T7
最佳 TRL-IRL 差距：+1 ~ +3

匹配逻辑：
• 企业 IRL=2（原型），导师 TRL=4~5 → ✅ 最佳
• 企业 IRL=3（试点），导师 TRL=5~6 → ✅ 最佳
• 企业 IRL=4（推广），导师 TRL=6~7 → ✅ 最佳
• 企业 IRL=2（原型），导师 TRL=8+ → ⚠️ 勉强（差距大）
• 企业 IRL=4（推广），导师 TRL=4 → ❌ 不匹配（太落后）

特殊规则：
• 技术型需求要求 TRL 差距精确（+1~+3 为最佳）
• 差距太大→沟通成本高；差距太小→缺乏引领
• 优先匹配有同类技术落地经验的导师
```

| 属性 | 值 |
|------|-----|
| 导师类型 | 技术型（架构师/技术专家） |
| 服务形式 | 技术方案、代码评审、技术攻关 |
| 周期 | 1-6个月 |
| 定价系数 | 2.0（技术深度） |

---

### 2.3 L2 工程型需求

**核心问题：** "方案都有了，需要人帮我们落地实施"

| 维度 | 定义 |
|------|------|
| **需求特征** | 系统实施、流程改造、项目管理、集成部署、培训交付 |
| **典型场景** | "模型有了，需要帮我们部署到产线" / "ERP需要升级，需要人带" |
| **企业 IRL** | L3-L5（试点→成熟） |
| **技术 TRL** | T7-T8（批量→规模化） |

**核心匹配规则：**

```
导师 TRL 最佳区间：T7-T8
最佳 TRL-IRL 差距：+1 ~ +2

匹配逻辑：
• 企业 IRL=3（试点），导师 TRL=5~6 → ✅ 最佳
• 企业 IRL=4（推广），导师 TRL=6~7 → ✅ 最佳
• 企业 IRL=5（成熟），导师 TRL=7~8 → ✅ 最佳
• 企业 IRL=3（试点），导师 TRL=9 → ⚠️ 勉强（实施≠研究）
• 企业 IRL=5（成熟），导师 TRL=5 → ❌ 不匹配（同级无引领）

特殊规则：
• 工程型需求更看重"同行业实施经验"而非学术水平
• TRL 差距不宜过大（企业需要的是"落地"不是"探索"）
• 优先匹配有同类企业实施案例的导师
```

| 属性 | 值 |
|------|-----|
| 导师类型 | 工程型（项目经理/实施顾问） |
| 服务形式 | 项目实施、系统集成、培训交付 |
| 周期 | 1-3个月 |
| 定价系数 | 1.2（标准化程度高） |

---

### 2.4 L1 服务型需求

**核心问题：** "系统在跑了，需要人帮我们运维优化"

| 维度 | 定义 |
|------|------|
| **需求特征** | 运维支持、故障排查、性能优化、版本升级、技术培训 |
| **典型场景** | "AI系统在跑了，偶尔有问题需要支持" / "数据库性能下降" |
| **企业 IRL** | L4-L5（推广→成熟） |
| **技术 TRL** | T9（产业化运营） |

**核心匹配规则：**

```
导师 TRL 最佳区间：T8-T9
最佳 TRL-IRL 差距：0 ~ +2

匹配逻辑：
• 企业 IRL=4（推广），导师 TRL=8~9 → ✅ 最佳
• 企业 IRL=5（成熟），导师 TRL=9 → ✅ 最佳（同级互助）
• 企业 IRL=5（成熟），导师 TRL=7 → ⚠️ 勉强（导师反而落后）
• 企业 IRL=4（推广），导师 TRL=5 → ❌ 不匹配（差太远）

特殊规则：
• 服务型需求允许 TRL 差距为 0（同级互助即可）
• 更看重"响应速度"和"同类系统运维经验"
• 可批量匹配（一个导师服务多家同类型企业）
```

| 属性 | 值 |
|------|-----|
| 导师类型 | 服务型（运维工程师/技术支持） |
| 服务形式 | 快速问答、远程支持、定期巡检 |
| 周期 | 按次/按月 |
| 定价系数 | 1.0（标准化+可替代性高） |

## 四、定价模型

### 4.1 定价公式

```
服务价格 = 基础价格 × 需求等级系数 × TRL稀缺性系数 × IRL差距系数
```

### 4.2 系数定义

**需求等级系数：**

| 等级 | 系数 | 理由 |
|------|------|------|
| L4 研究型 | 3.0 | 稀缺性高、不确定性大、价值创造潜力大 |
| L3 技术型 | 2.0 | 技术深度要求高、需要专业经验 |
| L2 工程型 | 1.2 | 标准化程度较高、可复制性强 |
| L1 服务型 | 1.0 | 基础服务、可替代性高 |

**TRL稀缺性系数：**

| TRL区间 | 系数 | 理由 |
|---------|------|------|
| T1-T3（学术型） | 1.5 | 学术导师稀缺、供给有限 |
| T4-T6（技术型） | 1.2 | 技术专家中等稀缺 |
| T7-T9（产业型） | 1.0 | 产业导师相对充足 |

**IRL差距系数：**

| TRL-IRL差距 | 系数 | 理由 |
|------------|------|------|
| +1~+2（最佳带教距离） | 1.0 | 标准服务 |
| 0（同级互助） | 0.8 | 打折（缺乏引领价值） |
| +3~+4（沟通成本高） | 1.3 | 溢价（需要更多精力翻译） |
| +5以上（不推荐） | 0.5 | 深度折扣（效果存疑） |
| 负数（导师太落后） | 0.3 | 不建议匹配 |

### 4.3 定价示例

```
场景1：L4研究型 + 学术导师(TRL=4) + 企业(IRL=1)
价格 = 基础价 × 3.0 × 1.5 × 1.0 = 4.5倍基础价

场景2：L3技术型 + 技术专家(TRL=6) + 企业(IRL=3)
价格 = 基础价 × 2.0 × 1.2 × 1.0 = 2.4倍基础价

场景3：L2工程型 + 实施顾问(TRL=7) + 企业(IRL=4)
价格 = 基础价 × 1.2 × 1.0 × 1.0 = 1.2倍基础价

场景4：L1服务型 + 运维工程师(TRL=9) + 企业(IRL=5)
价格 = 基础价 × 1.0 × 1.0 × 0.8 = 0.8倍基础价
```

## 六、匹配流程

### 6.1 完整匹配流程

```
第一步：需求诊断
  企业发布需求 → NLP分析 → 确定 A1-A6 定位 + L1-L4 等级
  企业填写 IRL 问卷 → 评估集成成熟度

第二步：处方生成
  根据 L1-L4 等级 → 确定导师最佳 TRL 区间
  根据 A1-A6 定位 → 确定技能匹配范围
  生成"匹配处方"：需要什么等级+什么技能的导师

第三步：配药执行
  在导师库中筛选 → TRL 区间过滤 → A1-A6 技能匹配 → 行业/规模/意愿过滤
  输出 Top 3 导师 + 可解释匹配理由

第四步：效果验证
  对接后跟踪 → 评价反馈 → 反馈到蒸馏模型和匹配算法
```

## 九、总结

### 一句话核心

> **L1-L4 决定"需要什么等级的导师"，TRL/IRL 决定"这个导师合不合适"，A1-A6 决定"这个导师有没有对应技能"。三者交叉，就是精准匹配。**

### 模型价值

1. **需求分级清晰**：企业知道自己的需求属于哪个等级
2. **匹配规则明确**：不是模糊的"合不合适"，而是精确的"差几级"
3. **定价逻辑透明**：每个系数都有明确的业务含义
