## 一、项目背景与目标

### 1.1 项目背景

企业数字化转型面临"技术找不着、人才匹配难、服务效果差"的困境。传统的导师推荐模式依赖人工经验，效率低、匹配精度差、难以规模化。

**核心痛点：**
- 技术供需信息不对称
- 导师能力评估缺乏量化标准
- 企业需求理解不精准
- 匹配效果难以衡量

### 1.2 项目目标

**LinkedEvery 智能体中台**致力于构建"技术转移的智能匹配引擎"，通过TRL/IRL双维度评估体系，实现导师与企业的精准匹配。

**核心目标：**
1. **精准匹配**：基于15个交叉点类型，实现导师-企业的精准匹配
2. **科学评估**：通过TRL/IRL九级评估，量化导师能力和企业需求
3. **智能推荐**：六维度匹配模型，多维度评估匹配质量
4. **商业闭环**：从匹配到服务到反馈的完整商业闭环

## 二、核心方法论：TRL/IRL框架

### 2.1 TRL（技术成熟度等级）

**定义**：衡量技术本身的成熟程度，从基础研究到产业化。

| 等级 | 定义 | 典型产出 |
|------|------|----------|
| T1 | 基础原理被观察和报告 | 学术论文、基础研究报告 |
| T2 | 技术概念和/或应用被提出 | 技术概念文档、可行性分析报告 |
| T3 | 概念验证（PoC） | 概念验证报告、实验室数据 |
| T4 | 实验室环境中的组件验证 | 实验室测试报告、组件验证数据 |
| T5 | 相关环境中的组件验证 | 相关环境测试报告、性能数据 |
| T6 | 相关环境中的系统/子系统模型演示 | 系统演示报告、集成测试数据 |
| T7 | 运行环境中的系统原型演示 | 运行环境演示报告、试运行数据 |
| T8 | 实际系统通过测试和验证完成 | 测试验证报告、量产准备文档 |
| T9 | 实际系统通过运行环境中的成功任务验证 | 运行验证报告、成功案例文档 |

### 2.2 IRL（集成成熟度等级）

**定义**：衡量企业将技术集成到现有系统/流程/组织中的能力。

| 等级 | 定义 | 典型场景 |
|------|------|----------|
| IRL 1 | 集成方案被识别 | 企业意识到需要引入AI质检，开始调研 |
| IRL 2 | 集成方案被定义 | 确定AI质检系统与现有MES/ERP的接口方式 |
| IRL 3 | 集成方案可行性被验证 | AI质检与MES系统在测试环境中成功对接 |
| IRL 4 | 集成技术被充分描述 | 完成接口文档、数据字典、异常处理规范 |
| IRL 5 | 集成方案在相关环境中被验证 | AI质检在模拟产线上运行，与MES实时同步 |
| IRL 6 | 集成方案在真实环境中被验证 | AI质检在客户工厂实际产线上运行1个月 |
| IRL 7 | 集成方案被完整集成 | AI质检成为产线的标准环节，自动触发/报警 |
| IRL 8 | 集成方案被验证为可靠 | AI质检系统运行6个月，准确率>99% |
| IRL 9 | 集成方案被证明为可持续 | AI质检系统已推广到所有产线，纳入质量管理体系 |

**匹配规则：**
- ✅ 最佳匹配（TRL - IRL = 1~2）：导师能带企业升级
- ⚠️ 可用匹配（TRL - IRL = 0 或 3~4）：同级互助或差距大
- ❌ 不匹配（TRL - IRL < 0 或 > 4）：导师太落后或太超前

## 四、六维度匹配模型

### 4.1 权重设计

| 维度 | 权重 | 说明 |
|------|------|------|
| **行业匹配度** | **30%** | 行业标签、子行业、产业链位置匹配（第一位） |
| **技能对齐度** | **20%** | 导师技能与企业需求主题、CDIO能力匹配（第二位） |
| **TRL/IRL 适配度** | **20%** | 与需求等级、最佳带教距离直接相关（第三位） |
| 规模适配度 | 10% | 企业规模与导师服务方式、组织复杂度匹配 |
| 服务形式 | 10% | 陪跑/咨询/培训/方案评审等服务形式匹配 |
| 地域时间 | 10% | 城市、通勤、响应周期、项目时长匹配 |

**验证**：前三维度权重 = 30% + 20% + 20% = 70% > 66.67% ✅

### 4.2 匹配得分计算

```python
def calculate_match_score(mentor, enterprise, requirement):
    """
    计算匹配得分（六维度）
    """
    # 1. 识别匹配类型
    match_type = identify_match_type(mentor.trl.level, enterprise.irl.level)
    
    # 2. 六维度评分
    industry_score = score_industry(mentor, enterprise)  # 30%
    skill_score = score_skill(mentor, enterprise, requirement)  # 20%
    trl_irl_score = score_trl_irl(mentor, enterprise, requirement)  # 20%
    scale_score = score_scale(mentor, enterprise)  # 10%
    service_score = score_service(mentor, enterprise, requirement)  # 10%
    geo_time_score = score_geo_time(mentor, enterprise)  # 10%
    
    # 3. 加权计算
    final_score = (
        0.30 * industry_score +
        0.20 * skill_score +
        0.20 * trl_irl_score +
        0.10 * scale_score +
        0.10 * service_score +
        0.10 * geo_time_score
    )
    
    return {
        'final_score': final_score,
        'match_type': match_type,
        'breakdown': {
            'industry': industry_score,
            'skill': skill_score,
            'trl_irl': trl_irl_score,
            'scale': scale_score,
            'service': service_score,
            'geo_time': geo_time_score
        }
    }
```

## 六、数据底座建设

### 6.1 数据模型

**核心实体：**
1. **导师画像**：TRL等级、技能标签、行业经验、成功案例
2. **企业画像**：IRL等级、行业标签、技术需求、预算规模
3. **匹配记录**：15类型、六维度评分、匹配理由、服务状态
4. **评估记录**：TRL/IRL评估历史、证据链、置信度

### 6.2 数据采集

**导师端数据采集：**
- 基础信息：姓名、职称、单位、研究方向
- TRL证据链：学术成果、专利成果、项目案例、产业化经验
- 技能标签：行业、技能、服务形式

**企业端数据采集：**
- 基础信息：企业名称、行业、规模、城市
- IRL评估问卷：18题，5维度评估
- 技术需求：需求描述、预算、周期、期望服务形式

### 6.3 数据质量

**质量控制机制：**
- 证据验证：论文/专利/项目自动验证
- 人工审核：关键数据人工复核
- 数据清洗：异常数据检测和修正
- 持续更新：定期复评，保持数据时效性
